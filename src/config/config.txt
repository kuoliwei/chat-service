========================================
Chat-Service 配置說明 (config.json)
========================================

{
  // ========================================
  // AI 相關設定
  // ========================================
  "ai": {

    // 對話上下文設定
    "contextWindow": {
      // 發送給 AI 時，最多包含的歷史訊息數
      // 例: 設為 10，則發送最近 10 條訊息作為對話歷史
      // 用途: 平衡上下文豐富度 vs 速度
      "maxMessages": 10,

      // 訊息中是否包含元數據（時間戳、ID 等）
      // 目前未實現，固定為 false
      "includeMetadata": false
    },

    // 各種操作的超時時間（單位: 毫秒）
    "timeouts": {
      // AI 生成回覆的最大等待時間
      // 超過時間會拋錯誤
      // 位置: serviceClient.js - generateResponse()
      "generateResponse": 30000,

      // RAG 初始化的最大等待時間
      // 建立新聊天室時的超時限制
      // ⚠️ 目前已移除此 timeout，改為無限等待
      // 位置: serviceClient.js - initializeRAG()
      "initializeRAG": 30000,

      // 清理 RAG 資料的最大等待時間
      // 刪除聊天室時的超時限制
      // 位置: serviceClient.js - cleanupRAG()
      "cleanupRAG": 10000
    }
  },

  // ========================================
  // RAG 檢索設定
  // ========================================
  "rag": {
    // 語義搜尋返回的結果數量
    // 檢索背景或 few-shot 範例時，最多返回 3 條最相關的結果
    // 值越大 = 資訊越多但檢索時間增加
    // 值越小 = 檢索快但可能丟失相關資訊
    "topK": 3,

    // 搜尋結果的相似度閾值 (0-1)
    // 只返回相似度 ≥ 0.7 的結果
    // 0.7 = 70% 相似度以上才算有用
    // 值越高 = 篩選越嚴格（只要最相關的）
    // 值越低 = 包含更多可能相關的結果
    "threshold": 0.7
  },

  // ========================================
  // 訊息摘要設定
  // ========================================
  "summary": {
    // 觸發摘要機制的字數閾值
    // 當對話歷史（除去最新 shortTermLimit 條訊息後）超過 500 字時，觸發摘要
    // 流程:
    //   1. 檢查舊訊息總字數 > 500？
    //   2. 如果是 → 將舊訊息摘要成 maxWords 字
    //   3. 保留最新 shortTermLimit 條訊息原文
    // 位置: conversationService.js - checkIfNeedsSummary()
    "threshold": 500,

    // 生成的摘要最多多少字
    // AI 生成摘要時，會被要求控制在 30 字以內
    // 位置: conversationService.js - triggerSummary()
    "maxWords": 30,

    // 摘要機制中不被摘要的最新訊息數
    // 保留最新 2 條訊息的原文，只摘要更早的訊息
    // 例: 有 10 條訊息 → 只摘要前 8 條，保留最新 2 條
    // 用途: 保留最近的聊天上下文完整性
    // 位置: conversationService.js - checkIfNeedsSummary()
    "shortTermLimit": 2,

    // 摘要相關的參數
    // 目前用途不明確
    "topK": 2
  }
}

========================================
典型場景示例
========================================

【場景 1】用戶發送新訊息
  1. 系統取最近 10 條訊息 (maxMessages)
  2. 檢查是否需要摘要 (threshold > 500?)
  3. 發送給 AI，超時 30 秒 (generateResponse)
  4. AI 回覆

【場景 2】對話過長（需要摘要）
  對話有 15 條訊息，總字數 800 字
  1. 檢查: 800 > 500? 是 → 需要摘要
  2. 保留最新 2 條訊息（原文）
  3. 摘要前 13 條訊息為 ≤30 字
  4. 新對話歷史 = 摘要(30字) + 訊息13號 + 訊息14號 + 訊息15號

【場景 3】RAG 檢索
  用戶說: "赫蘿，你昨天去哪？"
  1. 語義搜尋返回最多 3 條結果 (topK=3)
  2. 篩選相似度 ≥ 0.7 的結果 (threshold=0.7)
  3. 這些結果放入 AI 的 system prompt
  4. AI 用檢索結果和上下文生成回覆

========================================
修改建議
========================================

想提高 AI 回覆的上下文質量？
  → 增加 ai.contextWindow.maxMessages（如 15）

想加快回覆速度？
  → 減小 ai.contextWindow.maxMessages（如 5）

想提早觸發摘要？
  → 減小 summary.threshold（如 300）

想要 RAG 檢索更精準？
  → 增大 rag.threshold（如 0.8）

想要 RAG 檢索更寬泛？
  → 減小 rag.threshold（如 0.5）
